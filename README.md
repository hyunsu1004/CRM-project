## 📌 프로젝트 소개

투자 유치 과정에는 심사역, 팀장, 파트너 등 여러 이해관계자가 얽혀 있어 투자자 정보와 커뮤니케이션 기록이 여러 곳에 파편화되기 쉽습니다. 이로 인해 CEO 및 경영진은 어떤 투자자와 어떤 단계에서 어떤 대화를 나누고 있는지 한눈에 파악하기 어렵고, 기록을 정리하는 데 불필요한 시간과 에너지를 소모하게 됩니다.

**CLODGE CRM**은 다수의 스타트업이 투자 유치를 위해 VC(벤처캐피탈)와의 관계를 체계적으로 관리할 수 있도록 만든 CRM 플랫폼입니다. 투자자·기업 정보와 커뮤니케이션 히스토리를 하나의 소스로 단일화하고, 사용자가 원하는 속성을 자유롭게 추가할 수 있는 유연한 데이터 구조를 제공해 투자 데이터의 체계화와 투명성을 높이는 것을 목표로 합니다.

- 투자자 정보 및 딜(Deal) 진행 상황을 한 곳에서 관리
- 사용자가 원하는 커스텀 속성을 자유롭게 추가·관리 (EAV 구조)
- Selenium 크롤링으로 확보한 실제 투자 데이터 기반 정보 제공
- 딜 상세 화면에서 노트를 작성해 미팅·협상 이력을 기록

> 레퍼런스: [Attio](https://attio.com/), [Relate](https://www.relate.kr/)

## ✨ 핵심 기능

### 1. 메인 페이지
Top 10 스타트업 및 투자자를 실시간으로 보여줘 어떤 기업과 투자자에게 투자를 하고 받을 수 있는지 한눈에 파악할 수 있도록 구현했습니다. 최근 사용자 활동과 스타트업 뉴스도 함께 제공해 최신 투자 정세를 파악할 수 있습니다.

### 2. 기업 정보 상세 보기
검색량이 많은 Top 10 기업·투자자의 주요 카테고리, 총 투자금, 투자 단계, 최근 편당 일자 등 상세 정보를 모달로 확인할 수 있습니다.

### 3. 딜(회사·투자자) 정보 관리
- 딜 추가 시 회사명을 입력하면 기존 등록된 회사를 검색해 연결하거나, 없으면 즉시 신규 회사를 생성할 수 있습니다.
- 생성된 딜 정보는 AgGrid 기반 그리드 테이블에 나열되며 이름·생성일시·작성자 기준으로 정렬됩니다.
- 그리드 화면에서 속성 값을 바로 수정(editable)할 수 있습니다.

### 4. 커스텀 속성(EAV) 추가
회사명 같은 기본 정보 외에도 사용자가 원하는 정보를 **속성(Attribute)** 형태로 자유롭게 추가할 수 있습니다. 속성은 이름과 데이터 타입(Text, Number, Currency, Date, Select, Multi Select, Checkbox 등)을 가지며, 동일 이름의 속성은 허용하지 않고 Select/Multi Select 옵션 삭제 시 해당 옵션이 사용 중인지 검증하는 로직을 포함합니다.

### 5. 검색·정렬·필터링
투자자명 문자열 검색, 투자 라운드·투자 단계·투자 유형 등 조건부 필터링을 지원하며, Select/Person/Number·Currency/Date/DateRange 등 속성 타입별로 필터 UI가 다르게 제공됩니다.

### 6. 딜 상세 화면 & 노트
딜 목록에서 Row를 hover하면 나타나는 '열기' 버튼으로 상세 화면에 진입할 수 있고, 상세 화면에서는 노트를 작성·수정·삭제하며 미팅·협상 히스토리를 기록할 수 있습니다.

### 7. 거래 진행 상황 관리
My 딜 페이지를 통해 현재 진행 중인 거래와 각 거래의 상태(IR, 검토, 예비투자심의, 최종투자심의, 계약, 납입, 투자완료 등)를 한눈에 확인할 수 있습니다.

## 🏗️ 시스템 아키텍처

```
User → HTTPS/REST API → React SPA (Frontend)
                              ↕ HTTPS/REST API
                     Spring Boot (Application Server)
                     ├─ MySQL — EAV Schema (CRM Data & Deal States, 동적 속성 정의)
                     ├─ Gemini (Document Store) — 문서 분석, 감성 분석, 노트→EAV 데이터 추출
                     └─ Amazon S3 (Object Storage)

External Data Source: The VC (K-Startup Data) → Selenium Crawler (Background Job) → Spring Boot
```

- **React SPA**: 메인 대시보드, 기업 프로필, 딜 관리, 커스텀 속성 UI를 제공하는 프론트엔드
- **Spring Boot**: REST API 서버로 EAV 기반 CRM 데이터, 크롤링 데이터, AI 분석 결과를 조율하는 백엔드 코어
- **MySQL (EAV Schema)**: 회사·딜·노트 등 핵심 엔티티와 사용자 정의 속성을 저장하는 관계형 데이터베이스
- **Selenium Crawler**: The VC(K-Startup Data) 사이트에서 실제 투자 데이터를 크롤링·정제해 백엔드에 공급
- **Gemini**: 노트·문서를 분석해 감성 스코어링 및 EAV 데이터 추출에 활용
- **Amazon S3**: 첨부파일 및 문서 저장소

## 🐛 트러블 슈팅 — EAV 데이터베이스 구조

CLODGE CRM의 핵심 차별점은 **EAV(Entity-Attribute-Value) 모델**을 도입해 사용자가 원하는 속성을 동적으로 추가·관리할 수 있게 한 것입니다. 이 구조를 설계·구현하는 과정에서 다음과 같은 문제들을 마주하고 해결했습니다.

### 1) 고정 스키마의 한계
**문제**: 기존 CRM처럼 회사·딜 테이블에 컬럼을 고정해두면, 사용자마다 원하는 관리 항목(투자 라운드, 투자 단계, 담당 심사역, 커스텀 필드 등)이 달라 스키마 변경 없이는 새로운 항목을 추가할 수 없었습니다. 매번 컬럼을 추가/삭제하는 마이그레이션은 운영 중인 서비스에서 비용이 크고 위험했습니다.

**해결**: Company/Deal 같은 핵심 엔티티는 최소한의 고정 컬럼만 유지하고, 나머지 사용자 정의 정보는 **Attribute(속성 이름 + 데이터 타입) — Value(엔티티별 실제 값)** 구조로 분리했습니다. 이를 통해 스키마 변경 없이도 사용자가 원하는 시점에 텍스트·숫자·통화·날짜·단일/다중 선택·체크박스 등 다양한 타입의 속성을 자유롭게 추가할 수 있도록 구현했습니다.

### 2) 속성 타입별 값 저장 및 검증 로직 설계
**문제**: EAV 구조에서 모든 값을 문자열로만 저장하면 통화 계산, 날짜 정렬, 다중 선택 필터링 등에서 타입 안정성이 깨지는 문제가 있었습니다. 또한 Select/Multi Select 옵션을 삭제할 때 이미 해당 옵션을 사용 중인 데이터가 있는지 확인하지 않으면 참조 무결성이 깨질 위험이 있었습니다.

**해결**: 속성 생성 시 Data Type(Text, Number, Date, Currency, Select, Multi Select, Checkbox 등)을 명시적으로 지정하도록 하고, 값 저장·조회 로직에서 타입별로 분기 처리했습니다. 또한 동일 이름의 속성 중복 생성을 막고, Select/Multi Select의 옵션을 삭제하려 할 때 해당 옵션이 기존 값에 포함되어 있는지 사전에 체크해 경고(alert)를 띄우도록 해 데이터 정합성을 확보했습니다.

### 3) 동적 컬럼의 그리드(AgGrid) 렌더링 및 필터링
**문제**: 딜 목록을 그리드로 보여줄 때, 고정 컬럼과 달리 EAV로 저장된 사용자 정의 속성은 컬럼 개수와 타입이 사용자마다 달라 그리드 렌더링·정렬·필터링 로직을 일반화하기 어려웠습니다.

**해결**: AgGrid의 커스텀 컬럼 기능을 활용해 속성 메타데이터(이름, 타입)를 기준으로 컬럼을 동적으로 생성하고, 필터 UI 또한 속성 타입(Select/MultiSelect, Person, Number/Currency, Date, DateRange)에 따라 다른 컴포넌트를 렌더링하도록 분기 처리했습니다. 이를 통해 사용자가 추가한 속성이 몇 개든, 어떤 타입이든 그리드에서 검색·정렬·필터링·인라인 수정(editable)이 가능하도록 구현했습니다.

### 4) EAV 구조의 조회 성능 이슈
**문제**: 일반적인 EAV 패턴은 값 조회 시 여러 번의 조인이 필요해 데이터가 많아질수록 조회 성능이 저하될 수 있는 구조적 단점이 있습니다.

**해결**: 자주 조회되는 핵심 속성은 인덱스를 설계에 반영하고, 엔티티 단위로 속성 값을 함께 조회하는 쿼리 패턴을 정리해 불필요한 반복 조인을 줄였습니다. 이 과정에서 얻은 트레이드오프(스키마 유연성 ↔ 조회 성능)에 대한 이해를 바탕으로, 컬럼을 동적으로 관리하면서도 DB 공간 효율성을 높이는 방향으로 구조를 확정했습니다.
